# Chapter Contract

Choose chapters based on scope. Remove irrelevant sections instead of producing
empty boilerplate.

## Project Or Repository

1. Overview
2. Repository Map
3. Primary Workflows
4. Runtime And Tooling
5. Main Modules
6. Data Or Control Flow
7. Tests And Validation
8. Documentation, Scripts, And Config
9. Extension Points
10. Blind Spots

## Directory Or Package

1. Purpose And Ownership
2. Local File Map
3. Public Entry Points
4. Important Internal Flows
5. Dependencies And Callers
6. Tests And Validation
7. Common Edits
8. Blind Spots

## Module Or Feature Area

1. What It Does
2. User Or Developer Entry Points
3. Core Flow
4. Key Files And Types
5. Integration Points
6. Failure Modes
7. Tests And Debugging
8. Safe Change Checklist

## Documentation Chapter

1. Chapter Purpose
2. Main Claims
3. Source-Of-Truth References
4. Terms And Decisions
5. Related Files
6. Developer Takeaways
7. Open Questions Or Blind Spots

## Section Rules

- Use short section titles.
- Put file references close to the claim they support.
- Include commands only when they were found in source files or actually run.
- Separate facts from inferences with wording such as "The inspected files
  suggest..." when needed.
- Prefer a few high-signal sections over exhaustive directory listings.
